# DSLand

*(до 2026-07-20 проєкт звався "SIGame Team Quiz" — перейменовано на прохання dima, деталі в PROGRESS.md.
Сама тека на диску лишилась `SIGame-Team-Quiz` — це просто шлях, на назву гри він не впливає.)*

Командна вікторина у стилі "Своя гра", зроблена спеціально для ігрових вечорів dima з друзями.
Головна відмінність від класичної "Своєї гри": відповідають **команди по черзі** (round-robin), а не хто перший
натиснув кнопку.

Технології: Node.js + Express + Socket.IO (реальний час), фронтенд — звичайний HTML/CSS/JS без збірки,
дані — JSON-файли (без зовнішніх платних сервісів і без жодних API-ключів).

## Швидкий старт

**Потрібен Node.js** (LTS, з https://nodejs.org — звичайний інсталятор "Далі-Далі-Готово", ~2 хвилини).
Без нього нічого не запуститься (перевірте: у командному рядку `node --version` має щось показати, а не
помилку "не розпізнано").

Найпростіше — подвійний клік по `start-dsgame.bat` у корені проєкту (поруч із цим README): він сам зробить
`npm install` і `npm start`, вікно консолі лишиться відкритим, поки грає сервер. Якщо звичніше руками:

```
cd app
npm install
npm start
```

Сервер підніметься на порту 3000 (можна змінити в `app/.env`). У консолі одразу побачите готові посилання:

```
На цьому комп'ютері: http://localhost:3000
Для друзів у тій самій wifi-мережі:
  http://192.168.х.х:3000
```

### Як друзям підключитись
Усі мають бути в одній Wi-Fi/локальній мережі з комп'ютером, на якому запущено сервер. dima відкриває
`http://<IP-адреса-комп'ютера>:3000`, дає друзям цю саму адресу — вони вводять нікнейм і код кімнати (код
видає адмін-панель після створення кімнати) і потрапляють у гру. Кілька кімнат можуть існувати одночасно —
можна одразу грати двома компаніями на одному сервері.

### Адмін-панель
`http://<IP>:3000/admin.html` — пароль за замовчуванням `quiz2026`, встановлюється й змінюється в `app/.env`
(`ADMIN_PASSWORD=...`). Звичайні гравці не бачать і не можуть викликати адмінські дії навіть напряму через
API/сокети — сервер перевіряє токен адміна на кожен привілейований запит (деталі нижче, розділ "Безпека").

Кроки адміна: **Нова кімната** → скажіть друзям код → коли всі зайшли, **Сформувати команди** (можна
вказати кількість команд) → **Згенерувати теми** (одна кнопка — тягне свіжі, ще не використані теми з банку)
→ **Почати гру**. Під час гри доступні: перебалансування команд, форс-завершення питання, що зависло,
корекція відповіді (якщо автоматична перевірка помилилась), видалення гравця, дострокове завершення гри.

## Публічний доступ через інтернет (не лише локальна мережа)

Два варіанти, від найпростішого до постійного. **Обидва вимагають, щоб сервер уже був запущений**
(`start-dsgame.bat`) — тунель/деплой лише відкриває доступ до вже працюючого локального сервера ззовні.

### Варіант 1 — тимчасовий тунель (2 хвилини, без реєстрації)
Подвійний клік по `tunnel-localtunnel.bat` (або `tunnel-cloudflared.bat`, якщо перший не спрацює на вашій
мережі) у корені проєкту. У вікні консолі з'явиться публічне посилання (щось на кшталт
`https://xxxx.loca.lt` або `https://xxxx.trycloudflare.com`) — дайте його друзям, воно працює з будь-якого
пристрою в інтернеті, не лише вашого Wi-Fi. **Важливо:** посилання тимчасове — перестає працювати, щойно
закриєте те вікно консолі (або перезапустите комп'ютер). Не потребує жодного акаунту.

Обидва варіанти тунелю (localtunnel і cloudflared) перевірено наскрізно в еквівалентному середовищі 2026-07-20
— публічна адреса реально видається і запит через неї доходить до локального сервера. Це не було перевірено
саме на вашому комп'ютері (для цього потрібен ваш живий запуск), але сам механізм підтверджено робочим.

### Варіант 2 — постійний безкоштовний деплой (Render.com)
Для стабільної адреси, яка не залежить від вашого комп'ютера. Це вимагає вашої (dima) власної реєстрації —
жоден автономний запуск Claude не створює акаунти на сторонніх сервісах від вашого імені. Кроки:
1. Викладіть проєкт на GitHub (якщо ще не викладено) — потрібен звичайний git-репозиторій.
2. Зареєструйтесь на https://render.com (безкоштовний план достатній).
3. "New +" → "Blueprint" → підключіть репозиторій. Render сам підхопить `app/render.yaml` (там уже вказано
   `rootDir: app`, `npm install`, `npm start`, health-check на `/api/health`).
   Якщо оберете звичайний "Web Service" замість Blueprint — вручну вкажіть Root Directory `app`,
   Build Command `npm install`, Start Command `npm start`.
4. У розділі Environment Variables задайте `ADMIN_PASSWORD` і `SESSION_SECRET` (render.yaml навмисно не
   зберігає їх у git — задаються прямо в дашборді Render).
5. Деплой — Render видасть постійний URL на кшталт `https://dsland.onrender.com`.

Альтернатива Render — Railway (теж підтримує `app/Procfile` з `web: npm start`, і теж вимагає вказати
робочу директорію `app` в налаштуваннях сервісу).

### Чому зникають акаунти на Render (і як це виправити безкоштовно)
**Причина:** безкоштовний план Render не має постійного диска — при кожному редеплої, перезапуску чи
"засинанні" сервера через неактивність файлова система стирається й відновлюється рівно такою, якою вона
є в git-репозиторії. А `app/data/players.json`, `accounts.json`, `activity.json` і `usedThemes.json`
навмисно НЕ в git (це дані гравців, а не код — інакше кожен ваш push стирав би всі коіни всім) — тобто
після кожного такого скидання лишається тільки захардкоджений адмін-акаунт DimaSh, а всі інші акаунти й
коіни зникають. Код, що прив'язує коіни до акаунту, тут ні до чого — він і так правильний, це суто питання
хостингу.

**Виправлення (2026-07-22):** `src/state/remoteBackup.js` дублює ці 4 файли у безкоштовну Upstash Redis
базу через REST API (жодної нової npm-залежності, працює через вбудований у Node `fetch`) — при кожному
локальному записі файл асинхронно "підстраховується" в Upstash, а при старті сервера свіжа копія
підтягується назад НА диск ще до того, як будь-який модуль встигне щось прочитати. Без налаштування ця
функція просто вимкнена (сайт працює так само, як і зараз) — щоб увімкнути:
1. Зареєструйтесь на https://upstash.com (кредитна картка не потрібна для безкоштовного плану).
2. Console → Create Database → назвіть як завгодно (напр. `dsland-backup`), тип Regional, будь-який регіон.
3. На сторінці бази, вкладка REST API — скопіюйте `UPSTASH_REDIS_REST_URL` і `UPSTASH_REDIS_REST_TOKEN`.
4. У Render: ваш сервіс → Environment → додайте обидва значення під тими самими іменами.
5. Redeploy (або просто зачекайте наступний природний перезапуск) — у логах сервера має з'явитись
   "Бекап гравців/акаунтів у Upstash: увімкнено".

Безкоштовний план Upstash — 256 МБ і 500 000 команд/місяць, для гри в колі друзів це на порядки більше,
ніж будь-коли знадобиться.

## Архітектура (коротко)
- `server.js` — Express + Socket.IO вхідна точка.
- `src/state/roomManager.js` — ядро гри: кімнати, черга ходів команд, підрахунок очок, перевірка відповідей.
- `src/state/themeState.js` — банк тем + анти-повтор трекінг (`data/usedThemes.json`).
- `src/state/playersStore.js` — персистентна статистика гравців (`data/players.json`), правильні/неправильні відповіді.
- `src/logic/teamBalancer.js` — "змійка" (snake draft) розподіл гравців по командах за точністю.
- `src/logic/answerMatcher.js` — перевірка текстових відповідей (нечутлива до регістру, з допуском на дрібні одруки).
- `src/logo/logoGenerator.js` — процедурний генератор SVG-варіантів логотипів для механіки "Правильний логотип".
- `src/state/adminAuth.js` + перевірки в `src/socket/socketHandlers.js` — безпековий бар'єр адмін-функцій.
- `public/` — фронтенд (index/player/admin.html + vanilla JS).
- `data/themesBank.json` — згенерований банк із 61 теми × 5 питань (305 питань, станом на 2026-07-20). Скрипт,
  який його будує: `scripts/buildThemeBank.js` (можна перезапустити після редагування
  `data/handAuthoredThemes.part*.js`, якщо захочете дописати ще тем вручну).

## Голосовий чат команд
Кожна команда має свій ізольований голосовий канал: учасники команди чують одне одного, іншу команду —
ніколи. Технічно це mesh WebRTC (кожен клієнт тримає окреме з'єднання з кожним партнером по команді, звук
іде напряму між браузерами, через сервер лише проходить сигналізація) з новими Socket.IO подіями
`voice:join/leave/offer/answer/ice`. Ізоляція команд перевіряється на СЕРВЕРІ (`src/socket/socketHandlers.js`)
за реальним teamId гравця з `roomManager`, а не за тим, що надішле клієнт — тобто це не просто "чесна умова",
а реальний бар'єр. STUN-сервер — публічний `stun.l.google.com:19302`, без реєстрації; TURN-сервера немає
(теж вимагав би реєстрації/платного сервісу), тож у дуже суворих мережах (корпоративний Wi-Fi з жорстким
firewall) голос може не з'єднатися — для домашньої вечірки з друзями це не має бути проблемою. Кнопка
"Увімкнути мікрофон" з'являється в лобі (коли команди вже сформовані) і під час гри; якщо адмін
перебалансує команди, голосове з'єднання само розірветься й перевстановиться під нову команду.

## Нікнейм та аватарка гравця
Нікнейм гравець вказує на екрані входу (це вже було з першої версії). Новинка: там-таки можна одразу
завантажити власне фото як аватарку -- клік по кружку відкриває вибір файлу з телефону/компʼютера,
зображення стискається прямо в браузері (обрізається в квадрат, зменшується до 96×96, JPEG) перед
відправкою на сервер, тож у мережу йде лише кілька кілобайтів, а не оригінальне фото в кілька мегабайтів.
Аватарку можна змінити пізніше з лобі кнопкою "Змінити аватарку". Якщо фото не вибрано (або файл виявився
не зображенням чи завеликим), замість нього показується кружок з першою літерою ніка -- гра ніколи не
блокує вхід через погане фото, просто мовчки лишає заглушку. Аватарки видно в лобі (загальний список і
командні ростери), і в адмін-панелі поруч з кожним гравцем.

## Роль ведучого (адмін = ведучий)
Адмін-панель — це і є пульт ведучого, з правами як у "Своїй грі": окрім позначення останньої відповіді
правильною/неправильною (`Позначити ПРАВИЛЬНОЮ/НЕПРАВИЛЬНОЮ`), є окремий блок "Рахунок команд", де можна
довільно додати чи зняти будь-яку кількість балів будь-якій команді в будь-який момент гри — не лише
виправити останню відповідь. Додатково, для окремих гравців є легкий бонус-рахунок (кнопки +/- біля кожного
імені в списку гравців) — це НЕ те саме, що рахунок команди: команда й далі виграє гру своїм рахунком, а
особистий бонус — просто інструмент ведучого відзначити конкретну людину (наприклад, "найкраща відповідь
раунду"), не зберігається між іграми. Адмінка м'яко (не блокуюче) попереджає, якщо гравців менше 6 або
більше 14 — гру все одно можна почати.

Ведучий бачить суттєво більше, ніж гравці, і не лише останню відповідь:
- **Правильні відповіді заздалегідь** — згортавана панель "Відповіді" показує правильну відповідь/варіант
  КОЖНОГО питання поточного раунду ще до того, як команди його відкриють. Технічно це окремий канал
  `admin:room_state` (не той, що йде гравцям) — сервер в принципі ніколи не надсилає гравецькому сокету
  жодного поля з відповіддю, навіть для питань, які ще ніхто не відкривав.
- **Живий таймер** на панелі відкритого питання — та сама секундоміра, що бачать гравці, видно й ведучому,
  щоб не гадати "скільки ще команді думати".
- **Ручне перекидання гравця між командами** — випадаючий список біля кожного гравця в лобі (окремо від
  автоматичної "змійки", для точкових виправлень типу "цей друг прийшов пізно").
- **Музика ведучого** — поле для вставки посилання YouTube зліва на панелі + кнопка Play. Звук іде лише з
  колонок комп'ютера ведучого (не транслюється гравцям) — суто фонова музика між раундами, вмикається й
  вимикається вручну.

## Анімації та кастомний курсор
Плавні переходи на відкритті питання, підрахунку очок (пульс + спливаючі +/-N на пілі команди), ході
команди (індикатор світиться), hover/press кнопок і клітинок дошки, таймер червоніє й пульсує в останні
25% часу. Легка конфеті-анімація на правильну відповідь і на завершення гри (чистий CSS/DOM, без бібліотек).
Кастомний курсор — власний SVG у стилі палітри сайту (стрілка, а не абстрактна форма — щоб не постраждала
точність кліку), з окремим варіантом для клікабельних елементів; текстові поля навмисно лишили зі звичайним
текстовим курсором для зручності друку.

## Дрібні зручності інтерфейсу
- **"Як грати?"** — клік на заголовок будь-якої теми на дошці відкриває коротке спливаюче пояснення загальних
  правил (черга ходів, як рахуються бали) — не підказку по конкретній категорії, і не заважає клікам по самих
  клітинках цін.
- **Перехід між раундами** тепер супроводжується великим самозникаючим банером і в гравця, і в адмінки —
  раніше це відбувалось непомітно.
- **Вже відповідані клітинки** дошки більше не виглядають "порожніми" — приглушений фон і галочка замість
  зниклої ціни.
- **Кнопка "Цікавий факт"** у лобі й під час гри — короткий нейтральний факт на випадковий клік, щоб
  заповнити паузи очікування.

## Про лого
Коло з товстою малиновою обвідкою і тонким бірюзовим внутрішнім кільцем, великі жирні літери "DS" зверху,
під ними — кутастий силует Крампуса (роги, загострені вуха, важкі брови, "козячі" горизонтальні зіниці на
яскраво-оранжевих очах, вищирена паща з нерівними зубами й довгими іклами). Силует Крампуса має дуже темний
малиновий колір (`#2A0A10`) — це "тіньовий" відтінок малинового з палітри сайту (не 5-й колір, той самий
принцип, що й `--turquoise-ink`/`--shadow` в CSS), з тонкою оранжевою обвідкою для "жаристого" ефекту.
`public/img/logo.svg` — повна версія (у шапках сторінок), `public/favicon.svg` — спрощена, обрізана на
роги+очі+ікла версія для вкладки браузера (при 16-32px деталі літер/тексту однаково нечитабельні, тому
залишили лише найконтрастніший фрагмент).

## Про автогенератор тем — важливе пояснення дизайн-рішення
У додатку свідомо **немає** виклику зовнішнього LLM API під час гри — за умовою немає жодного API-ключа від
dima, і додаток має працювати повністю офлайн/локально без платних сервісів. Тому "генератор тем" влаштований
так: наперед (мною, Claude, під час цієї збірки) написано банк із 61 теми у стилі оригінального пака dima
(дивіться `reference/theme-reference.md`) — поп-культура/ігри/меми/спорт/історія/мультфільми/їжа/прапори/
слогани, і серед них 4 теми механіки "Правильний логотип" генеруються процедурно (SVG, без повторів картинок).
Кнопка "Згенерувати теми" в адмінці бере випадкові ще не використані теми з цього банку і чесно веде облік
використаного (`data/usedThemes.json`), щоб теми не повторювались між іграми — навіть після перезапуску
сервера. Якщо банк колись вичерпається (тепер це десь після ~6 повних ігор поспіль, банк зростав тричі),
система м'яко переходить на повторне використання найдавніше показаних
тем (а не ламається), і адмін-панель про це попереджає. Банк легко розширити: додайте нові об'єкти в
`data/handAuthoredThemes.part1.js` / `part2.js` і перезапустіть `node scripts/buildThemeBank.js`.

Якщо dima захоче правжній безлімітний живий генератор — найпростіший наступний крок: додати підтримку
опційного `ANTHROPIC_API_KEY` в `.env` і невеликий модуль, який (якщо ключ є) звертається до Claude API за
новою темою, а якщо ключа немає — як і зараз, працює з банку. Це не зроблено зараз (нуль-конфіг за умовою),
але банк спроєктовано так, щоб такий модуль підключити було легко.

## Про 4-колірну палітру
Використано рівно 4 кольори: білий, бірюзовий, малиновий (crimson), оранжевий. Для читабельності основного
тексту та легких тіней використано ТЕМНІ/СВІТЛІ ВІДТІНКИ цих самих 4 кольорів (наприклад, темно-бірюзовий
для тексту замість чорного) — це не новий, п'ятий колір, а той самий бірюзовий у іншій яскравості; так само
тінь елементів — напівпрозорий чорний лише для глибини/об'єму, це технічна тінь, а не колір дизайну. Якщо
dima мав на увазі буквально нуль відтінків — скажіть, і це легко зробити суворішим. Той самий принцип
використано для силуету Крампуса на лого (дуже темний малиновий, розділ "Про лого" нижче) і для 4-ї/5-ї
команди на дошці (темний відтінок 1-го/2-го кольору команди замість повторення того самого відтінку).

Свідомий виняток — **аватарки** (як завантажене фото, так і нові готові варіанти, розділ нижче): це "контент",
а не елемент інтерфейсу, тому вони можуть бути будь-якого кольору, так само як фото-аватарка завжди могла.

## Спрощення, про які варто знати
- **Ідентифікація гравця = нікнейм** (за замовчуванням без пароля, як і на самому початку — "доступ відкритий
  для всіх"). Це означає, що статистика правильних/неправильних відповідей прив'язана до нікнейму: якщо двоє
  друзів зайдуть з однаковим нікнеймом в різний час, вони "поділять" одну статистику. Для вечірки з друзями
  це не проблема, просто майте на увазі.
- **Акаунти (2026-07-21) — опційний шар поверх того самого нікнейма.** У "Особистому кабінеті" можна лишити
  поле пароля порожнім (стара поведінка, без змін) або вписати пароль — тоді для цього нікнейма створюється
  (або перевіряється) акаунт: логін = нікнейм, пароль хешується через `crypto.scrypt` (ніколи не зберігається
  відкритим текстом), сесія — токен у `localStorage`, як і в адмін-панелі. Один-єдиний акаунт має роль
  `admin` — `DimaSh` (пароль задає dima) — і лише він бачить кнопку "Адмін-панель" на сторінці вікторини та
  панель видачі ККоїнів у своєму кабінеті; це саме перевіряється і на сервері (`authSessions.requireAdmin`),
  а не лише ховається в інтерфейсі. Деталі — `src/state/accountsStore.js` та `src/routes/authRoutes.js`.
- Рахунок команди: правильна відповідь = **+ціна питання**, неправильна/тайм-аут = **−ціна питання**
  (класичне правило "Своєї гри"). Легко вимкнути мінусування — прапорець `NEGATIVE_ON_WRONG` у
  `src/config.js`.
- Гравець, що приєднався ПІСЛЯ того, як адмін уже сформував команди (або лишився без команди після кіку),
  не потрапляє автоматично в жодну команду до наступного перебалансування (перебалансувати можна лише в
  лобі/після гри, не посеред раунду — свідоме обмеження, щоб не зламати рахунок і чергу ходів). Такий
  гравець бачить чітке пояснення на екрані ("Без команди" / "попросіть адміна перебалансувати") замість
  мовчки заблокованої дошки.
- Час на відповідь команди: 45 секунд (`ANSWER_TIMEOUT_MS` у `src/config.js`), потім автоматичний тайм-аут
  і перехід ходу.

## Тестування
`npm install`/живий запуск сервера цього разу реально перевірено — на відміну від попередніх сесій, sandbox
цього запуску дав доступ до registry.npmjs.org: `npm install` пройшов, сервер піднявся і відповів на
`/api/health`, а обидва варіанти тунелю (`tunnel-localtunnel.bat`/`tunnel-cloudflared.bat`) видали робочі
публічні адреси (деталі — `PROGRESS.md`, розділ про четвертий запуск). Це перевірено в тимчасовій пісочниці
Claude, не на самому комп'ютері dima — але це найсильніше підтвердження того, що механізм справді робочий,
за всю історію проєкту. Окремо від цього, усю ігрову логіку й далі перевірено напряму через Node без
зовнішніх залежностей:
- юніт-тести чистих модулів (перевірка відповідей, "змійка" розподілу команд, генератор логотипів);
- повний інтеграційний прогін через справжній `socketHandlers.js`/`roomManager.js` із саморобною
  заглушкою Socket.IO API (`scripts/fakeSocketHarness.js`, `scripts/integrationTest.js`) — **132/132**
  перевірок проходять (було 73/73 на початку, +33, +13, +18 у наступних сесіях): логін адміна, захист
  адмін-подій від гравців, приєднання гравців, формування команд, генерація дошки, повний прогін гри
  питання-за-питанням (обидва типи питань — текст і вибір лого), перехід ходів по черзі, підрахунок очок,
  завершення раунду/гри, перепідключення гравця зі збереженням команди, ручна корекція відповіді адміном,
  форс-завершення "завислого" питання, кік гравця, регресійні тести на реальні знайдені баги (зависання при
  кіку активної команди, відновлення підказки після перезаходу), ручне коригування рахунку команди й гравця,
  повний сценарій голосового чату, аватарки гравців, ручне перекидання гравця між командами (лобі-only,
  security-перевірка), і межа безпеки `publicState()`/`adminState()` (відповідь ніколи не потрапляє гравцю,
  адмін отримує її разом із живим таймером через окремий канал `admin:room_state`);
- `scripts/extraChecks.js` — окремий незалежний прогін перевірок безпосередньо на РЕАЛЬНОМУ згенерованому
  банку тем: дедуп усіх SVG-варіантів логотипів (жоден з 30 логотипних питань не має двох однакових
  варіантів), відсутність "протікання" правильної відповіді в даних, що йдуть до клієнта, узгодженість
  цінової сітки, крайні випадки перевірки текстових відповідей, реальний розподіл "змійкою" на тестовому
  наборі гравців, і розмір банку тем (мінімум 60 тем/300 питань + присутність кожної нової категорії).
- Детально — у `PROGRESS.md` (включно з окремим звітом про повторну незалежну перевірку).

Рекомендація: перед вечіркою запустіть `start-dsgame.bat` (або вручну `npm install && npm start`) і зробіть
один швидкий тестовий раунд удвох/утрьох, щоб на живих клієнтах (браузер) усе виглядало так, як задумано.
Це й досі важливо: попри те, що серверна й тунельна логіка тепер перевірені найретельніше за всю історію
проєкту (сандбокс-прогони, 132/132 тестів), жодна автономна сесія так і не бачила саму гру в реальному
браузері на вашому комп'ютері — це єдине, що може підтвердити лише жива людина.
